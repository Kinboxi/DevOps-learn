# Сети — база

> [!info] Зачем это знать
> Сети — фундамент DevOps. Без понимания IP, DNS, TCP/UDP невозможно траблшутить серверы, настраивать nginx, работать с облаком.

---

## IP адрес

Каждое устройство в сети имеет свой адрес — IP адрес. Без него данные не знают куда идти.

Два типа:
- **Локальный** — внутри твоей сети (`192.168.x.x`, `10.x.x.x`)
- **Публичный** — уникальный в интернете (`142.250.74.46`)

> [!tip] Аналогия
> Локальный IP — номер комнаты в квартире. Публичный — адрес самого дома.

### ip a
```
> показывает все сетевые интерфейсы и их IP адреса
```

### ifconfig
```
> альтернатива ip a, показывает интерфейсы и адреса
```

---

## MAC адрес

Физический адрес устройства, вшит в железо. Не меняется.

- IP — логический адрес, может меняться
- MAC — физический адрес, постоянный

> [!tip] Аналогия
> MAC — паспорт. IP — прописка.

MAC работает **внутри локальной сети**. IP работает **между сетями**.

---

## DHCP

Протокол который автоматически выдаёт IP адреса устройствам в сети. Без DHCP пришлось бы каждому устройству вручную прописывать IP.

> [!tip] Аналогия
> Роутер = администратор в отеле. Ты подключился — он выдал тебе номер комнаты. Отключился — номер освободился.

У выданного IP есть **lease time** — время аренды. После истечения адрес может смениться.

---

## Маска подсети

Показывает где в IP адресе заканчивается сеть и начинается устройство.

```
IP:    192.168.0.101
Маска: 255.255.255.0
```

Где в маске `255` — это сеть. Где `0` — это устройство.

```
192.168.0  → адрес сети
101        → твоё устройство
```

## DNS

Переводит доменное имя в IP адрес. Без DNS пришлось бы писать `142.250.74.46` вместо `google.com`.

> [!tip] Аналогия
> DNS — телефонная книга интернета. Ты называешь имя — получаешь номер.

Популярные публичные DNS:
```
8.8.8.8    — Google
1.1.1.1    — Cloudflare
```

### ping google.com
```
> проверяет доступность хоста и показывает его IP адрес
```

## Порты

IP адрес — адрес устройства. Порт — адрес конкретной программы на этом устройстве.

```
192.168.1.10:22    — SSH
192.168.1.10:80    — HTTP
192.168.1.10:443   — HTTPS
```

> [!tip] Аналогия
> IP — адрес здания. Порт — номер квартиры.

| Порт | Протокол   |
|------|------------|
| 22   | SSH        |
| 80   | HTTP       |
| 443  | HTTPS      |
| 53   | DNS        |
| 3306 | MySQL      |
| 5432 | PostgreSQL |
| 6379 | Redis      |

## Модель OSI

7 уровней которые описывают как данные путешествуют по сети. Нужна для траблшутинга — ищешь проблему снизу вверх.

| Уровень | Название       | Пример        |
|---------|----------------|---------------|
| 7       | Приложение     | HTTP, SSH, DNS|
| 6       | Представление  | TLS           |
| 5       | Сеанс          |               |
| 4       | Транспорт      | TCP, UDP      |
| 3       | Сеть           | IP            |
| 2       | Канальный      | MAC, Ethernet |
| 1       | Физический     | кабель, WiFi  |

> [!warning] Важно
> Уровни 3, 4, 7 — встречаются в DevOps чаще всего.

Сайт не открывается — идёшь снизу вверх:
```
1. Есть ли соединение? (уровень 1)
2. Пингуется ли IP? (уровень 3)
3. Открыт ли порт? (уровень 4)
4. Работает ли приложение? (уровень 7)
```

## TCP vs UDP

Оба на уровне 4. Два разных способа доставки данных.

| | TCP | UDP |
|--|-----|-----|
| Надёжность | да, подтверждает доставку | нет |
| Скорость | медленнее | быстрее |
| Используется | SSH, HTTP, файлы | видеозвонки, игры, DNS |

> [!tip] Аналогия
> TCP — заказное письмо с уведомлением. UDP — бросил в ящик и забыл.

## HTTP / HTTPS

Протокол по которому браузер общается с сервером. Запрос → ответ.

Методы:
- **GET** — получить данные (открыл страницу)
- **POST** — отправить данные (заполнил форму)

| Код | Значение              |
|-----|-----------------------|
| 200 | OK, всё хорошо        |
| 301 | перенаправление       |
| 404 | страница не найдена   |
| 500 | ошибка сервера        |

**HTTPS** — тот же HTTP но зашифрованный через TLS. Порт 443.

### curl url
```
> отправляет HTTP запрос и показывает ответ сервера
```

### curl -I url
```
> показывает только заголовки ответа (код, тип контента и тд)
```

### curl -v url
```
> подробный вывод — весь процесс запроса и ответа
```

## TLS / SSL

Протокол шифрования. Уровень 6 OSI. Делает HTTP → HTTPS.

Как работает TLS Handshake:
1. Клиент подключается к серверу
2. Сервер отдаёт сертификат (доказывает что он настоящий)
3. Договариваются о методе шифрования
4. Всё дальнейшее общение зашифровано

> [!info]
> SSL — устаревшее название. Сейчас везде TLS, но говорят SSL по привычке.

## SSH

Протокол для безопасного удалённого подключения к серверу. Порт 22, TCP.

> [!info] Зачем
> Ты дома — сервер в дата-центре. SSH позволяет управлять им как будто сидишь рядом.

### ssh user@ip
```
> подключиться к серверу
```

### ssh -p 2222 user@ip
```
> подключиться если SSH на нестандартном порту
```

### ssh-keygen
```
> генерирует пару ключей (публичный и приватный) для входа без пароля
```

### ssh-copy-id user@ip
```
> копирует публичный ключ на сервер — после этого вход без пароля
```

Конфиг SSH сервера:
```
/etc/ssh/sshd_config
```

## ARP

Протокол который переводит IP адрес в MAC адрес внутри локальной сети. Когда роутер знает IP получателя — ARP находит его MAC чтобы доставить пакет физически.

> [!tip] Аналогия
> Знаешь номер квартиры (IP) но не знаешь как выглядит дверь (MAC). ARP — спросить у соседей.

### arp -a
```
> показывает ARP таблицу — список IP и MAC адресов устройств в сети
```

---

## ICMP

Протокол для диагностики сети. Не передаёт данные — только проверяет доступность и сообщает об ошибках. Именно его использует `ping`.

### ping ip
```
> проверяет доступность хоста и измеряет время ответа
```

### ping -c 4 google.com
```
> отправить ровно 4 пакета и остановиться
```

## Утилиты траблшутинга

### ping ip
```
> проверяет доступность хоста (уровень 3 OSI)
```

### traceroute ip
```
> показывает весь путь пакета до хоста — через какие роутеры проходит
```

### mtr ip
```
> комбинация ping и traceroute в реальном времени
```

### ss -tuln
```
> показывает все открытые порты
```

### ss -tulnp
```
> то же самое но с именами процессов (нужен root)
```

### netstat -tuln
```
> альтернатива ss, старая утилита — показывает открытые порты
```

### telnet ip port
```
> проверяет открыт ли конкретный порт на хосте
```

### nc -zv ip port
```
> netcat — проверяет доступность порта, удобнее telnet
```

### nmap ip
```
> сканирует открытые порты на хосте
```

### nmap -sV ip
```
> сканирует порты и определяет версии сервисов
```

### tcpdump -i eth0
```
> захватывает сетевые пакеты на интерфейсе для анализа трафика
```

### tcpdump -i eth0 port 80
```
> захватывает только трафик на порту 80
```

### iftop
```
> показывает сетевой трафик в реальном времени — кто сколько потребляет
```

## Nginx

Веб сервер и обратный прокси. Принимает запросы и распределяет их между серверами приложения.

### systemctl status nginx
```
> проверить статус nginx
```

### systemctl restart nginx
```
> перезапустить nginx
```

### nginx -t
```
> проверить конфиг на ошибки перед перезапуском
```

### nginx -s reload
```
> перечитать конфиг без полного перезапуска
```

Конфиг и логи:
```
/etc/nginx/nginx.conf
/etc/nginx/sites-available/
/var/log/nginx/access.log
/var/log/nginx/error.log
```

## HAProxy

Балансировщик нагрузки. Распределяет трафик между несколькими серверами.

> [!info] Отличие от Nginx
> HAProxy заточен именно под балансировку — гибче для этой задачи. Nginx универсальный.

### systemctl status haproxy
```
> проверить статус haproxy
```

### haproxy -c -f /etc/haproxy/haproxy.cfg
```
> проверить конфиг на ошибки
```

## iptables

Файрвол Linux. Управляет каким трафиком разрешить, заблокировать или перенаправить.

### iptables -L
```
> показать все текущие правила файрвола
```

### iptables -L -n -v
```
> подробный вывод с счётчиками пакетов
```

### iptables -A INPUT -p tcp --dport 80 -j ACCEPT
```
> разрешить входящий трафик на порт 80
```

### iptables -A INPUT -p tcp --dport 22 -j ACCEPT
```
> разрешить SSH подключения
```

### iptables -A INPUT -j DROP
```
> заблокировать весь остальной входящий трафик
```

> [!warning] Осторожно
> Неправильное правило может заблокировать SSH и ты потеряешь доступ к серверу. Проверяй дважды.

## Что происходит когда открываешь сайт

1. Вводишь `google.com` в браузер
2. DNS переводит `google.com` → IP `142.250.74.46`
3. Браузер устанавливает TCP соединение (порт 443)
4. TLS Handshake — договариваются о шифровании
5. Браузер отправляет GET запрос
6. Сервер отвечает HTML (код 200)
7. Браузер отображает страницу

> [!info]
> Всё это происходит за доли секунды.
